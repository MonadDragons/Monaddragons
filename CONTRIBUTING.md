# Monad Dragons Contributing Guide

## Building the Living Dragon Protocol Together

Version: Genesis Foundation v1.0

Status: Public Contributor Framework

---

# 1. Purpose

This document defines how contributors participate in the Monad Dragons ecosystem.

Monad Dragons is an open-source living protocol.

Every contribution becomes part of the Dragon evolution.

Contributors are not only writing code.

They are helping build:

- Dragon systems
- Dragon knowledge
- Dragon communities
- Dragon infrastructure

Every contribution must respect the Dragon Codex.

The Codex is the foundation.

---

# 2. Contribution Philosophy

Monad Dragons follows five principles:

## Respect The Foundation

Every contribution must align with:

- DragonCodex.md
- Privacy principles
- Security principles
- Open-source values

---

## Build With Purpose

Every feature must answer:

Why does this strengthen the Dragon?

Unnecessary complexity should be removed.

---

## Documentation Is Code

Documentation is part of the protocol.

Every meaningful implementation must update related documentation.

A feature without documentation is incomplete.

---

## Security Before Speed

Never introduce:

- Private keys
- Secrets
- Credentials
- Sensitive user information

into the repository.

Security is permanent.

---

## Evolution Together

The Dragon grows through collective contribution.

Quality is more important than quantity.

---

# 5. Development Rules

All contributions should follow:

## Minimal Design

Prefer simple solutions.

Complexity must have purpose.

---

## Modular Architecture

Systems should remain independent and replaceable.

---

## Clear Naming

Names should represent their purpose.

Dragon terminology should be preferred.

Examples:

Use:

- Dragon
- Dragon Oracle
- Dragon Channel
- Dragon Knowledge

Avoid unnecessary generic terminology.

---

## Consistent Documentation

Every new system should include:

- Purpose
- Architecture
- Usage
- Limitations

---

# 7. Pull Request Principles

Every Pull Request should include:

## Description

Explain:

- What changed
- Why it changed
- How it strengthens the Dragon

---

## Documentation

Update related files.

Examples:

Code change:

Update:

```
ARCHITECTURE.md
```

Dragon behavior change:

Update:

```
DRAGONS.md
```

Protocol rule change:

Update:

```
DragonCodex.md
```

---

## Review

Contributions are reviewed for:

- Alignment
- Security
- Quality
- Simplicity
- Documentation

---

# 8. Dragon Constitution Compliance

No contribution may violate:

- Privacy First
- Blockchain principles
- Immutable supply rules
- Security standards
- User control principles

The Dragon Codex overrides implementation preference.

---

# 9. Security Rules

Never commit:

```
Private keys

Seed phrases

Passwords

API keys

Authentication tokens

User private data
```

Use secure external systems for sensitive information.

Public repositories must remain safe.

---

# 10. Documentation Style

Documentation should be:

## Minimal

Remove unnecessary words.

---

## Clear

Every sentence should carry meaning.

---

## Timeless

Avoid temporary assumptions.

---

## Protocol Focused

Write like infrastructure documentation.

Not marketing material.

---

# 11. Code Quality

Code should be:

- Readable
- Tested
- Maintainable
- Documented

Prefer long-term stability over short-term speed.
